Ticket 8716 · Microlearning — Test Strategy

TestRail for test-case management & traceability · Playwright for web E2E automation · scope derived from PLAN.html (build-day storyboard, Waves 1–4 + multi-language thread) · owners @john.demendoza / @chaitanya.thakur

Scope in one line: Validate the committed Microlearning slice — admin enables the module, builds a Topic, adds Link/PDF/Video-URL Activities, grants access By Location, and a Learner opens each activity so the topic rolls up 2/3 → 3/3 → Completed — across 5 languages, on team2.staging.bissafety.app.
Never-cut core loop (= P0): create topic → add activity → learner opens → completion tracked. Everything else is prioritised below it.
Platform boundary: Playwright covers web (admin + learner). The Safetapp React Native thread is out of Playwright's reach — tracked in TestRail, executed manually (Appium/Detox noted as future work, not this effort).

Priority: P0 core loop / gating P1 full feature P2 enhancement / edge  |  Coverage: Auto Playwright Hybrid auto + manual check Manual human only

1 · How the two tools split the work

ToolOwnsDoes not own
TestRail Single source of truth for test cases, priorities, and coverage; organises cases into Suite → Sections; groups execution into Milestones / Test Plans / Runs; records manual results; receives automated results via API; reports pass rate & traceability to tickets (W1-1 … C19). Does not run tests. Not a place to store step-by-step automation code.
Playwright Executes browser E2E flows for the web admin + learner apps; seeds/cleans data via bisapi; runs smoke on every PR and full regression before the demo; produces trace/video/screenshot artifacts on failure. Does not author or prioritise cases (that lives in TestRail). Does not test Safetapp RN mobile.

The bridge: every automated Playwright test is annotated with its TestRail case ID (e.g. C1042); a CI reporter opens a Run in TestRail and posts pass/fail back — so one coverage report shows manual + automated together. See §5.

4 · Playwright automation design

Project layout

  • @playwright/test + TypeScript. tests/ (specs by section) · pages/ (Page Objects) · fixtures/ (auth + seeded data) · data/ (topic/activity builders) · utils/ (API client, TestRail reporter).
  • Auth once per role: global setup logs in an admin and a learner test account, saves storageState; specs reuse it — no repeated UI login. Credentials come from CI secrets / env, never hardcoded.
  • Seed via API, assert via UI: topics/activities/permissions are created through bisapi wrappers (saveTopic, getMyTopics, getTopic, recordCompletion) in fixtures for speed & determinism; the UI is driven only for the behaviour under test. Each spec cleans up what it seeds.

Selector strategy

  • Prefer getByRole + getByTestId over CSS/text. Request devs add data-testid on: topic card, pillbox, activity type icon, Start button, progress counter, language chip.
  • Known gap: the drag-reorder handle is div role="button", not a native <button> (per W4-4). It needs an explicit data-testid to be reliably targetable — and that non-native control is itself an accessibility finding to log.

Tricky behaviours & how to automate them

BehaviourPlaywright approach
Link / PDF open in new tabcontext.waitForEvent('page') → assert the new tab's URL / target; PDF rendering stays a manual check.
Video URL embedAssert the embed iframe is present with the expected src; playback quality is manual.
Live progress rollupOpen activity → assert counter transitions 2/3 → 3/3 and status flips to Completed without reload.
Completion idempotencyFire recordCompletion twice (UI re-open + direct API) → assert progress capped and count stable.
By-Location scopingSeed two learners in different locations; assert visibility differs; All-Locations flips both to visible.
fr-CA localizationSeparate Playwright project with a French learner profile; run core specs; assert French labels resolve + default-language fallback; fail on leaked English strings.

Tagging & run selection

  • Grep tags: @smoke @p0 @regression @localization @wave1…@wave4.
  • @smoke = the never-cut loop only — the fast gate that must stay green.

CI integration

  • Every PR into team2-hackathon: run @smoke against staging — blocks merge if red.
  • Pre-demo / nightly: full regression incl. fr-CA project.
  • On failure: attach trace, video, screenshot. Retries = 1 (surface flakes, don't hide them).

5 · Traceability — Playwright ↔ TestRail

  • Each automated spec title carries its case ID, e.g. test('C1042 · learner completes topic 3/3', …), or a Playwright annotation of type testrail.
  • CI creates a Run inside the milestone's Test Plan and a reporter posts each result (pass/fail + trace URL) to the matching case via the TestRail API.
  • When a case is automated, set its Automation Status → Automated and fill Playwright Ref. Un-automated cases stay Candidate or Manual — the report shows the real automated-vs-manual split, no blind spots.
  • Both directions covered: TestRail case → Refs → storyboard ticket (W2-1) → PM case; and Playwright test → case ID → section → priority.

MCP note: a testrail connector is available in this workspace (list_suites, list_sections, list_runs, fetch_run_cases…) — usable later to script section/case scaffolding or pull live run status once the project exists.

7 · Risk focus, environments & rollout

Highest-risk areas (deepen coverage here)

  • Completion loop & idempotency — it's the committed demo; double-recording or a stuck rollup is the worst failure. Most automation weight here.
  • Multi-language / localization — storyboard says "QA watches this thread hardest"; fallback + no-hardcoded-strings + fr-CA are its own risk-based suite.
  • By-Location permissions — a scoping bug leaks a topic to the wrong learners; test positive + negative visibility explicitly.
  • Module toggle gating — OFF must fully hide the feature for both roles.

Environment & data

  • Target: team2.staging.bissafety.app (HTTPS). Web admin + web learner.
  • Roles: System admin (enable module, topic/activity CRUD, permissions) · Learner (with a Location + Language profile).
  • Data matrix: ≥2 Locations (for By-Location + All-Locations) · 5 supported languages with fr-CA as the mandatory walkthrough · one topic per pillbox state (Active / Draft / Deactivated).
  • Test accounts only; secrets injected via CI — no real credentials handled in tests.

Entry / exit criteria (per wave)

  • Entry: CONTRACTS frozen, endpoints reachable (stub ok), testCommand skeleton green.
  • Exit: all P0 cases pass on staging · @smoke automation green · no open Sev-1 · fr-CA run executed.

Automation rollout (mirrors the build order)

  • Phase A — framework + auth/data fixtures + @smoke skeleton (aligns with P0-5 testCommand).
  • Phase B — Admin topic + activity CRUD specs (Waves 1–2).
  • Phase C — Learner completion loop + idempotency (Wave 3) — the demo guardrail.
  • Phase D — Localization + By-Location + Wave-4 regression, layered on once the core is green.
