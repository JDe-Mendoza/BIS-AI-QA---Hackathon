A · Product decisions to confirm (with our recommended answer)

G1productRole/permission matrix — how many roles do we enforce?
The spec names many roles (Senior Client Admin, Client Security Admin, Client Admin, Course Admin, plus BIS network roles). Building the full matrix is days of work. Recommended: enforce a single admin role gate (Client Admin can do everything) and a single learner account; provision the schema for the full matrix but do not build per-role gating.
G2productTitle limit — "50 alphanumeric characters" (line 108).
"Alphanumeric" literally excludes spaces and punctuation, which would reject "Defensive Driving 101." Recommended: treat as 50 characters allowing spaces + basic punctuation, with a live counter. Confirm intent.
G3productVideo completion — full-watch detection vs open-to-complete.
Spec (line 1035) says "possible to detect if a user watched the whole video." True detection needs the player APIs and event wiring, a rabbit hole. Recommended: v1 marks a Link/PDF/Video complete when the learner opens/launches it (spec already defines Link + PDF this way). Video full-watch is a fast-follow.
G4productActivity types in the dropdown for v1.
Spec offers Video, Link, PDF, and Activity (SCORM ZIP). SCORM and uploaded MP4 are out of scope. Recommended: v1 dropdown = Link, PDF, Video URL (YouTube/Vimeo). Hide "Activity" (SCORM) and MP4-upload.
G5productTopic Settings page — how much do we build?
The full Settings page is huge (General, CEUs, Languages, Status, Public Access). We only need the path that makes the learner side work. Recommended: build only Settings → Permissions (By Location). Skip CEUs, Languages, Public Access, Status dropdown for v1.
G6productSingle language only for v1.
Multi-language (per-language uploads, auto-translate, "use default for all languages") is deferred, but the Language control is woven through Create Topic and every activity. Recommended: v1 uses the portal default language only; hide the "add another language" UI entirely.
G7product"Complete Again" on activity edit.
Drives re-completion and progress recalculation (lines 272-298), genuinely complex. Recommended: v1 edit saves fields only; no Complete Again (unchecked behavior = users unaware, which the spec allows).
G8productPermissions table columns without events.
The Managing-Permissions table is mostly event columns (Event Status/Start/End). We deferred events. Recommended: show Name / Type / summary, with Event Status = N/A for all rows.

B · Technical gotchas & risks (we mitigate)

G9devFile storage for PDF (and images) is the single biggest technical risk.
PDF upload (<15MB) and image upload (<4MB) both need a real file store. If the dev environment does not expose the existing BIS upload service, this blocks the PDF activity type. Mitigation / fallback: confirm the existing upload pattern early (codegraph + anchor files). Fallback if hard: ship Link + Video URL only, demo PDF via a link to a hosted PDF. Decide the trigger by ~10:30.
G10devCompletion-detection for links/PDFs that open in a new tab.
The activity opens in a new browser tab; completion must still be recorded, and cannot come from the new tab. Mitigation: frontend calls a recordCompletion(activityId) endpoint at the click-to-open (learner already authenticated in the portal). Define this in the CP-2 contract explicitly.
G11devCFML verification loop is slower than React.
Frontend builds/tests locally in a worktree; CFML changes need the dev environment to verify, so the backend lane's lap is longer. Mitigation: backend leans on TDD (tests run without the full app) + a smoke script; plan merge cadence around the CFML lap length.
G12devCustom Date Format dependency (lines 309, 341).
"Last Updated" is meant to follow the portal's custom date-format setting. Mitigation: v1 uses one sensible default date format; do not build the custom-date-format setting. Note as a known simplification.
G13devTopic is Active on creation but invisible until a permission exists.
Spec: "default Status is Active… no permissions have been granted yet." Not a bug, but it will confuse the demo if forgotten. Mitigation: bake the demo path in: create topic (Active) → add activities → add By-Location permission → learner sees it. Put this in the demo script.

D · Clean-cut boundaries (deferred features inside v1 screens)

In the spec's dashboard/settings UI but out of scope. Hide or grey cleanly so the UI does not dangle broken buttons or imply network behavior:
  • Company drop-down on the dashboard (line 50), a multi-portal switcher. Remove for v1.
  • Training Provider + "Launched" status on cards (lines 79-86), a network concept. Show portal name only; no launched state.
  • Public Access / Launch to Companies / Decommissioning, the entire network-sharing surface. Not rendered.
  • Draft & Deactivated topic states, v1 topics are effectively Active-only (Deactivate/Reactivate is a stretch). Show Active only, or render the pillbox with empty Draft/Deactivated.
  • Event/notification affordances (Schedule Event toggle, Event Notification editor, email log), deferred with events. Not rendered.
  • CEUs toggle, Auto-translate, Push to Subportals. Not rendered.

E · The one mechanic to design carefully: completion tracking

This is the spine of the demo, so worth a dedicated contract. The microlearning_completion record should capture userId, activityId, topicId, firstCompletedAt, lastCompletedAt, status (Not Started / In Progress / Completed). Provision the two timestamps now even though v1 only sets completion on open, so a later "Complete Again" fast-follow needs no migration. Topic progress = count(Completed) / count(active activities); topic status rolls up New / In Progress / Completed. Define all of this in CP-2 so backend and the learner UI build against the same shape.
